# RTC-Tools × pymoca benchmark — results

Comparison of pymoca **0.9.2** (RTC-Tools' current pin, `== 0.9.*`) against the
**`fix-inherited-symbol-scope-pr`** branch **plus two fixes on this branch**:
(1) honoring `MODELICAPATH`, (2) preserving `input`/`output` causality for
alias-typed variables. Driving RTC-Tools 2.8.0's example suite through pymoca's
CasADi backend. Both run in virtualenvs with an identical solver stack (casadi
3.7.2, numpy, scipy); only the pymoca version differs. The PR run has
`MODELICAPATH=<checkout>/test/libraries/MSL-4.0.x`.

Reproduce with `tools/rtc_tools_benchmark/run_benchmark.sh` (see `README.md`).
Tables below the `---` are generated by `compare.py`; this top section is the
human read. The full pymoca test suite (`pytest test/ --ignore=test/msl_examples_test.py`)
passes with both fixes applied: **449 passed, 10 skipped, 53 xfailed, 0 failed**.

## Fix 1 — honor MODELICAPATH (`src/pymoca/backends/casadi/api.py`)

The Deltares models `extend Modelica.Icons.Package`, an MSL class that was not
on pymoca's compile search path. `_compile_model` now merges any `MODELICAPATH`
roots as a lazy tree, so MSL classes resolve.

**Result: the `ModelicaSemanticError: Extends name Modelica.Icons.Package not
found` crash is eliminated for every example.** Scripts passing under the PR
branch went from **3/18 → 10/18** — the fix resolves the reported error.

Getting past the MSL crash exposed further, pre-existing PR-branch bugs
(masked before because flattening aborted early).

## Fix 2 — preserve input/output for alias-typed variables (`src/pymoca/tree/_flattening.py`)

Of the 10 scripts unblocked by fix 1, several (`basic`, `single_reservoir`)
exported **only the `time` column** — their `input`/`output` signals were
missing entirely.

Root cause: RTC-Tools' variables are typed with MSL alias types
(`Modelica.Units.SI.VolumeFlowRate = Real(...)`). When such a derived type
collapses to its builtin leaf during flattening, the outer variable's prefixes
are copied onto the leaf — except the code explicitly excluded `input`/`output`
(`_flattening.py`, was ~line 279):

```python
for pfx in flat_symbol.prefixes:
    if pfx not in ("input", "output") and pfx not in inner_sym.prefixes:
        inner_sym.prefixes.insert(0, pfx)
```

Minimal reproduction (no MSL, no Deltares):

```modelica
package P
  type Flow = Real(unit="m3/s");
  model M
    input Flow u;  output Flow y;
  equation y = u; end M;
end P;
```

| flattened symbol | 0.9.2 | PR branch (before fix 2) | PR branch (after fix 2) |
| --- | --- | --- | --- |
| `u` | `prefixes=['input']` | `prefixes=[]` | `prefixes=['input']` |
| `y` | `prefixes=['output']` | `prefixes=[]` | `prefixes=['output']` |

The exclusion was meant for connector-local causality, which is already
stripped up-front for *nested* symbols (same file, the `if prefix:` guard
around line 191). For a top-level variable the prefix should propagate to the
leaf; the fix removes the `input`/`output` exclusion from the outer-prefix
propagation loop.

**Result:** `basic` and `single_reservoir` now export the correct columns
(`Q_in`, `Q_release`, `V_storage`, …), and the flattened model *fingerprint*
(variable classification into inputs/outputs/states/alg_states) now matches
0.9.2 almost exactly. Full test suite still green (449 passed, 0 failed).

## What's still wrong (separate, pre-existing PR-branch bugs — not fixed here)

Columns now match, but **values don't always match**, and 8/18 scripts still
fail. Both are traced to the same underlying issue, distinct from causality:

### Zero-length array dimensions not resolved

`basic`'s flattened model keeps an extra algebraic variable,
`storage.QForcing`, that 0.9.2 correctly has zero of. The model declares:

```modelica
parameter Integer n_QForcing(min = 0) = 0;
input Modelica.Units.SI.VolumeFlowRate QForcing[n_QForcing];
```

`n_QForcing` defaults to `0`, so `QForcing` should flatten to a zero-length
array (no variable at all) — that's what 0.9.2 does. On the PR branch, the
dimension expression is never substituted with the resolved value of
`n_QForcing`; it stays an unresolved `Primary(value=None)`, so instead of
vanishing, `QForcing` materializes as one spurious free variable. That gives
the optimizer an unconstrained extra degree of freedom in the mass balance
(`der(V) = QIn - QOut + sum(QForcing)`), which lets it satisfy the equations
without moving `Q_release` — hence `basic`, `lookup_table`,
`pumped_hydropower_system`, and `single_reservoir` all run but produce wrong
numbers (`Q_release` stuck near 0, `max rel diff = 1.0`).

The same array-dimension-resolution gap is almost certainly what produces the
`ValueError: Symbol …H_b was given an index of 1 but this symbol is not an
array` failures in `cascading_channels`, `channel_pulse`, and both
`channel_wave_damping` scripts — a parametrically-sized array not being
resolved before indexing.

## Compile time

The MSL now loads on every cold compile, so pymoca time rises from ~1.5–3 s
(0.9.2) to ~17–20 s per model on the PR branch — dominated by parsing MSL
package files while building the lazy MODELICAPATH tree. A like-for-like time
comparison is only meaningful once the examples produce equivalent results.

## Bottom line

Both fixes are correct, resolve what they target, and don't regress the pymoca
test suite. RTC-Tools compatibility is meaningfully closer (3/18 → 10/18
scripts run; columns now match where they run) but not complete: the parametric
/ zero-length array-dimension resolution bug is the next blocker, affecting
both the numerical correctness of the 10 passing scripts and the 4 array-index
failures.

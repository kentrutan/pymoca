# RTC-Tools × pymoca benchmark — results

Comparison of pymoca **0.9.2** (the version RTC-Tools currently pins,
`pymoca >= 0.9.1, == 0.9.*`) against the **`fix-inherited-symbol-scope-pr`**
branch (`origin` tip `f36ad5a`), driving RTC-Tools 2.8.0's example suite through
pymoca's CasADi backend. Both were run in virtualenvs with an **identical**
solver stack (casadi 3.7.2, numpy, scipy); only the pymoca version differed.

Reproduce with `tools/rtc_tools_benchmark/run_benchmark.sh` (see `README.md`).
The tables below the `---` are generated by `compare.py`; this top section is
the human read of them.

## Bottom line

**The PR branch is not backward-compatible with RTC-Tools as it stands.**
17 of 18 example scripts flatten and run under 0.9.2; only **3 of 18** run under
the PR branch, and 2 of those 3 don't actually invoke pymoca. Every RTC-Tools
model built on the `Deltares` Modelica library — i.e. essentially the whole
suite — fails to flatten.

- **0.9.2**: 18/18 scripts succeed.
- **PR branch**: 3/18 succeed. The three are the two `channel_wave_damping`
  helper *modules* (no `transfer_model` call) and the one self-contained model
  (`simulation_with_custom_equations`) that does not touch the Deltares library.
- Where both versions *do* compile the same model
  (`simulation_with_custom_equations`), the **flattened structure is identical**
  and the **numerical results are bit-identical** (max diff `0.0`). So the
  backend contract itself is intact — the regression is purely in name
  resolution during flattening.

## Root cause

Every failure is the same error, raised from the PR branch's rewritten
name-lookup / instantiation code:

```
pymoca.tree.ModelicaSemanticError: Extends name Modelica.Icons.Package not found in scope Deltares
```

(a second variant reports `... not found in scope Modelica.Units`).

- The `Deltares` library ships by `rtc-tools-channel-flow`; its
  `Deltares/package.mo` begins with `extends Modelica.Icons.Package` — a purely
  **cosmetic** Modelica Standard Library (MSL) base class used only for icon
  annotations.
- `rtc-tools-standard-library` ships only a **partial** MSL: `Modelica/Units.mo`
  exists, but there is **no `Modelica.Icons` package** on the library path. And
  `Units.mo` *itself* opens with `extends Modelica.Icons.Package`,
  `extends Modelica.Icons.Information`, etc.
- **pymoca 0.9.2 tolerates these unresolved cosmetic `extends`** — an
  unresolvable class reference during flattening is caught and skipped
  (`except (KeyError, ast.ClassNotFoundError): pass`), so models flatten fine
  even though `Modelica.Icons.*` is never provided.
- **The PR branch made `extends` resolution strict**: an `extends` target that
  cannot be found in scope raises `ModelicaSemanticError`
  (`tree/_instantiation.py:_find_extends_class`). Because RTC-Tools relies on
  the tolerant behaviour, every Deltares-based model now aborts.

This is a genuine downstream-compatibility regression, not a harness or library
misconfiguration: the library paths passed to pymoca are identical for both
versions (they come from RTC-Tools' own `modelica_mixin`), and only the pymoca
version changes the outcome.

## Compile time

Not meaningfully comparable: the PR branch never reaches a successful cold
flatten for any Deltares model, so there is no baseline-vs-candidate time for
the suite. The single model that compiles under both
(`simulation_with_custom_equations`) took 1.87 s under 0.9.2 vs 2.95 s under the
PR branch — but that is one tiny model on a first cold run (the PR branch also
initialises its new SQLite parse-cache DB on first use), so it is not a reliable
performance signal. A useful time comparison requires the flatten failures to be
resolved first.

## Recommendation

To make the PR branch usable by RTC-Tools, the strict `extends` resolution needs
a compatibility path for unresolved cosmetic MSL base classes — either restore
the tolerant "skip unresolvable `extends`" behaviour (optionally behind a
compiler option) or provide a stub `Modelica.Icons` package. Once models
flatten again, re-run this benchmark to get the real compile-time and
numerical-result comparison across the full suite.

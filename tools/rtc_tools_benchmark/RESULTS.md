# RTC-Tools × pymoca benchmark — results

Comparison of pymoca **0.9.2** (RTC-Tools' current pin, `== 0.9.*`) against the
**`fix-inherited-symbol-scope-pr`** branch **plus the MODELICAPATH fix on this
branch**, driving RTC-Tools 2.8.0's example suite through pymoca's CasADi
backend. Both run in virtualenvs with an identical solver stack (casadi 3.7.2,
numpy, scipy); only the pymoca version differs. The PR run has
`MODELICAPATH=<checkout>/test/libraries/MSL-4.0.x`.

Reproduce with `tools/rtc_tools_benchmark/run_benchmark.sh` (see `README.md`).
Tables below the `---` are generated by `compare.py`; this top section is the
human read.

## What the MODELICAPATH fix does

The Deltares models `extend Modelica.Icons.Package`, an MSL class that was not on
pymoca's compile search path. `_compile_model` now merges any `MODELICAPATH`
roots as a lazy tree (`src/pymoca/backends/casadi/api.py`), so MSL classes
resolve.

**Result: the `ModelicaSemanticError: Extends name Modelica.Icons.Package not
found` crash is eliminated for every example.** Scripts passing under the PR
branch went from **3/18 → 10/18**. So the fix does exactly what was asked — it
resolves the specific RTC-Tools error.

## But the fix is necessary, not sufficient

Getting past the MSL crash exposes **several deeper, pre-existing PR-branch
bugs** (previously masked because flattening aborted early). Of the 10 scripts
that now run, only a few produce results equal to 0.9.2; the rest either export
degenerate output, produce different numbers, or fail later in translation:

### 1. `input`/`output` causality dropped for alias-typed variables (root-caused)

`basic` and `single_reservoir` now run but export **only the `time` column** —
their control/output signals (`Q_release`, `V_storage`, …) are missing.

RTC-Tools declares these as `input`/`output` variables typed with **MSL alias
types** (`Modelica.Units.SI.VolumeFlowRate = Real(...)`). The PR branch drops the
`input`/`output` prefix whenever the variable's type is a derived/alias type.
Minimal reproduction (no MSL, no Deltares involved):

```modelica
package P
  type Flow = Real(unit="m3/s");
  model M
    input Flow u;  output Flow y;
  equation y = u; end M;
end P;
```

| flattened symbol | 0.9.2 | PR branch |
| --- | --- | --- |
| `u` | `prefixes=['input']` | `prefixes=[]` |
| `y` | `prefixes=['output']` | `prefixes=[]` |

Plain `Real`-typed `input`/`output` variables are preserved correctly by both.
Root cause: `src/pymoca/tree/_flattening.py` (~line 279). When a derived type
collapses to its builtin leaf, the outer variable's prefixes are copied onto the
leaf **except** `input`/`output`:

```python
for pfx in flat_symbol.prefixes:
    if pfx not in ("input", "output") and pfx not in inner_sym.prefixes:
        inner_sym.prefixes.insert(0, pfx)
```

The exclusion is meant for connector-local causality, which is already stripped
up-front (same file, ~line 191, only for *nested* symbols where `prefix` is set).
For a top-level variable the prefix survives and should propagate to the leaf,
but this line drops it. This is independent of the MODELICAPATH fix and is the
single biggest reason RTC-Tools output is still wrong.

### 2. Different numerical results

`lookup_table` (max rel diff **0.53**) and `pumped_hydropower_system` (rel diff
**1.0**) run but produce values that diverge from 0.9.2 — not just missing
columns. Same family of variable-classification differences feeding the solver.

### 3. New translation/solve failures (8 scripts)

| error | examples |
| --- | --- |
| `ValueError: Symbol …H_b was given an index of 1 but this symbol is not an array` | cascading_channels, channel_pulse, channel_wave_damping (×2) |
| `NotImplementedError: Wrong number or type of arguments for overloaded function 'GenDM_ones'` | goal_programming, mixed_integer |
| `RuntimeError: … ImplicitToNlp::init for 'next_state'` | integrator_delay, simulation |

These are array-dimension handling, CasADi `ones()` generation, and ill-posed
system (a downstream symptom of the causality loss) — each a separate PR-branch
issue, out of scope for the MODELICAPATH change.

## Compile time

The MSL now loads on every cold compile, so pymoca time rises from ~1.5–3 s
(0.9.2) to ~17–20 s per model on the PR branch — dominated by parsing MSL
package files while building the lazy MODELICAPATH tree. A like-for-like time
comparison is only meaningful once the examples produce equivalent results.

## Bottom line

The MODELICAPATH fix is correct and resolves the reported error. Making RTC-Tools
fully work on the PR branch additionally requires fixing the surfaced
flatten/translate bugs — starting with the `input`/`output` prefix drop at
`_flattening.py:279`, which is small and well-isolated.
